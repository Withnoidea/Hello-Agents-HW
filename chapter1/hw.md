3. 某电商公司正在考虑两种方案来处理售后退款申请：

方案 A （Workflow）：设计一套固定流程，例如：

A.1 对于一般商品且在 7 天之内，金额 < 100RMB 自动通过；100–500RMB 由客服审核；>500RMB 需主管审批；而特殊商品（如定制品）一律拒绝退款

A.2 对于超过 7 天的商品，无论金额，只能由客服审核或主管审批；

方案 B （Agent）：搭建一个智能体系统，让它理解退款政策、分析用户历史行为、评估商品状况，并自主决策是否批准退款

- 这两种方案各自的优缺点是什么？

  | -------- | ------------------------------ | -------------------------------- |
  | 决策方式 | 按照预先定义的规则执行         | 根据目标自主分析和决策           |
  | 灵活性   | 较低，规则固定                 | 较高，可适应复杂场景             |
  | 可解释性 | 很强，每一步都有明确规则       | 相对较弱，决策过程可能较复杂     |
  | 开发成本 | 较低，实现简单                 | 较高，需要模型训练、工具调用等   |
  | 运行成本 | 低                             | 较高，需要 AI 推理计算           |
  | 稳定性   | 高，不容易出现异常             | 存在幻觉、误判等风险             |
  | 适用任务 | 重复性、规则明确的业务         | 复杂、开放式、需要综合分析的业务 |

- 在什么情况下 Workflow 更合适？什么情况下 Agent 更有优势？如果你是该电商公司的负责人，你更倾向于采用哪种方案？

- ## Workflow 更适合的情况

  Workflow 更适用于：

  业务规则明确且稳定；

  决策逻辑简单；

  对稳定性和可解释性要求较高；

  每天需要处理大量重复业务。

  例如：

  7 天无理由退款；

  自动审批优惠券；

  自动发货；

  自动退款金额计算；

  库存扣减等。

  这些任务几乎没有推理需求，因此 Workflow 效率最高。

  ------

  ## Agent 更适合的情况

  Agent 更适用于：

  业务规则复杂；

  需要综合多个信息来源；

  存在大量例外情况；

  需要推理、规划或个性化决策。

  例如：

  判断退款是否存在欺诈风险；

  分析用户是否恶意退款；

  判断商品是否影响二次销售；

  VIP 用户补偿方案制定；

  客诉处理与安抚。

  这些任务很难完全用固定规则描述，因此 Agent 更具优势。

  ## 如果我是该电商公司的负责人，我**不会完全采用 Workflow 或完全采用 Agent，而是更倾向于采用两者结合的方案**。

  原因如下：

  - 对于大部分规则明确的退款申请，Workflow 能够快速、稳定地完成审批，降低系统成本，提高处理效率。
  - 对于复杂、特殊或高风险的退款申请，再交由 Agent 进行综合分析，可以提高决策质量和用户满意度。
  - 最终对于 Agent 仍无法确定的情况，再由人工客服或主管审核，从而兼顾效率、准确性和安全性。

- 是否存在一个方案 C，能够结合两种方案，达到扬长避短的效果？

- 完全可以设计一个结合两者优势的**混合方案（Hybrid Workflow-Agent）**。

  例如：

  ```
  用户提交退款申请
          │
          ▼
  Workflow规则判断
          │
   ┌──────┴────────┐
   │               │
  规则明确       规则复杂
   │               │
  自动审批      Agent分析
   │               │
   │        风险评分、信用评估
   │               │
   └──────┬────────┘
          ▼
  需要人工？
     │         │
   否         是
   │          │
  自动完成    客服/主管审批
  ```

  ### 混合方案的优势

  1. **提高处理效率**

     大量简单退款可直接自动审批，仅少量复杂案例交给 Agent 或人工处理。

  2. **降低运行成本**

     Agent 只处理复杂任务，不必对所有退款申请进行 AI 推理，减少计算资源消耗。

  3. **提高决策准确率**

     Workflow 保证规则执行的一致性，Agent 提供复杂场景下的智能分析，两者互补。

  4. **增强系统稳定性**

     即使 Agent 出现误判，也可以通过 Workflow 限制其决策范围，并设置人工审核作为最终保障，降低业务风险。

5. 卡尼曼的“系统 1”（快速直觉）和“系统 2”（慢速推理）理论[2]为神经符号主义 AI 提供了很好的类比。请首先构思一个具体的智能体的落地应用场景，然后说明场景中的：

- 哪些任务应该由“系统 1”处理？
- 哪些任务应该由“系统 2”处理？
- 这两个系统如何协同工作以达成最终目标？

下面给出一个完整的示例，以**医疗诊断助手**为应用场景。

------

## 医疗诊断助手中的“系统1”和“系统2”协同工作

### （1）应用场景

设计一个医院智能医疗诊断助手，辅助医生进行疾病诊断。患者输入症状，并上传血常规、CT、MRI 等检查结果，智能体快速分析病情，提供初步诊断建议，并结合医学知识库生成诊断依据和治疗建议，最终辅助医生完成诊断。

------

### （2）哪些任务由“系统1”处理（快速直觉）

系统1对应神经网络驱动的亚符号主义 AI，擅长模式识别和快速判断。

在该场景中主要负责：

- **识别医学影像**：快速识别 CT、MRI、X 光片中的病灶、肿瘤、肺炎等异常区域。
- **分析生理指标**：根据血压、心率、血常规等数据快速判断是否存在异常。
- **理解患者自然语言描述**：识别患者描述的症状，如“发烧、咳嗽、胸闷”等，并提取关键信息。
- **初步疾病预测**：根据大量历史病例，快速预测可能的疾病类型及其概率。

系统1的特点是速度快、能够处理海量非结构化数据，但其判断主要基于统计学习，缺乏严格的逻辑推理能力。

------

### （3）哪些任务由“系统2”处理（慢速推理）

系统2对应符号主义 AI，擅长逻辑推理和规则分析。

在该场景中主要负责：

- **依据医学指南进行推理**：结合临床诊疗规范和医学知识库，对系统1的预测结果进行验证。
- **综合多种检查结果**：分析影像、化验指标、既往病史等信息之间的关系，排除矛盾或不合理的诊断。
- **生成诊断依据**：说明为什么判断为某种疾病，并引用相关医学知识或临床指南。
- **制定治疗方案**：根据疾病类型、患者年龄、过敏史等因素推荐合理的治疗方案，并提示可能存在的风险。

系统2虽然推理速度较慢，但具有较好的可解释性和可靠性，适合高风险医疗决策。

------

### （4）两个系统如何协同工作

医疗诊断助手采用神经符号主义 AI，将系统1和系统2结合起来完成诊断任务，其协同流程如下：

```text
患者输入症状、检查结果
          │
          ▼
系统1（快速直觉）
- 理解自然语言
- 识别医学影像
- 分析化验数据
- 预测可能疾病
          │
          ▼
系统2（逻辑推理）
- 查询医学知识库
- 根据诊疗指南验证结果
- 综合患者病史进行推理
- 生成诊断依据和治疗建议
          │
          ▼
输出最终诊断建议
```

例如，当患者出现“持续发烧、咳嗽、胸痛”等症状时，系统1可快速分析胸部 CT 图像，预测患者可能患有肺炎；随后，系统2结合血常规、既往病史及临床诊疗指南进行逻辑推理，验证诊断结果，并给出推荐治疗方案及相应依据。

通过这种协同方式，系统1负责快速感知和模式识别，系统2负责逻辑推理和可解释决策，两者优势互补，既提高了诊断效率，又增强了诊断结果的准确性和可信度。

6. 尽管大语言模型驱动的智能体系统展现出了强大的能力，但它们仍然存在诸多局限。请分析以下问题：

- 为什么智能体或智能体系统有时会产生“幻觉”（生成看似合理但实际错误的信息）？
- 智能体产生“幻觉”（Hallucination）是指生成了**看似合理但实际上错误、不存在或无法验证的信息**。其主要原因包括以下几个方面：

  1. **大语言模型的生成机制决定了其本质是概率预测。**
      LLM 的目标是预测下一个最可能出现的词，而不是验证事实，因此当训练数据不足或知识缺失时，模型可能根据已有模式“合理猜测”，从而生成错误信息。
  2. **训练数据存在局限性。**
      模型的知识来源于训练数据，若数据存在错误、过时或覆盖不足，模型就可能学习到错误信息，或者无法回答最新发生的事件。
  3. **缺乏真实世界验证能力。**
      模型本身没有真正理解现实世界，也不能自动验证答案是否正确。如果没有调用搜索引擎、数据库等外部工具，仅依赖内部知识容易产生幻觉。
  4. **复杂推理过程中误差累积。**
      智能体通常需要多轮推理和工具调用，如果某一步推理出现错误，后续步骤可能继续基于错误信息进行分析，最终导致错误结论。

  因此，即使智能体具有较强的语言生成能力，也不能保证所有回答都真实可靠，在医疗、法律、金融等高风险场景通常需要外部知识库或人工审核。

- 在 1.3 节的案例中，我们设置了最大循环次数为 5 次，如果没有这个限制，智能体可能会陷入什么问题？
- 在 1.3 节的旅行助手案例中，程序设置了：

  ```
  for i in range(5):
  ```

  如果没有这一限制，智能体可能出现以下问题：

  ### ① 无限循环（Infinite Loop）

  模型可能一直认为：

  > 还需要继续思考。

  例如：

  ```
  Thought
  ↓
  
  Action：查询天气
  
  ↓
  
  Observation
  
  ↓
  
  Thought：再查询一次天气
  ```

  不断重复调用同一个工具，永远无法结束任务。

  ------

  ### ② 工具调用死循环

  例如：

  ```
  查询天气
  ↓
  
  查询景点
  
  ↓
  
  重新查询天气
  
  ↓
  
  重新查询景点
  
  ↓
  
  ……
  ```

  智能体不断重复调用工具，却始终无法进入 `Finish` 状态。

  ------

  ### ③ 成本不断增加

  每一次循环都会：

  - 调用 LLM
  - 调用 API
  - 消耗 Token
  - 增加运行时间

  无限循环会导致计算资源和 API 调用费用不断增加。

  ------

  ### ④ 错误不断传播

  如果第一次 Observation 就有错误，例如天气查询失败，智能体可能一直围绕错误信息反复推理，形成"错误反馈循环"，最终仍无法得到正确答案。

  因此，在实际 Agent 系统中通常都会设置：

  - 最大循环次数（Max Iterations）
  - 最大 Token 数
  - 超时时间（Timeout）
  - 重复 Action 检测

  以保证系统能够安全终止。

- 如何评估一个智能体的“智能”程度？仅使用准确率指标是否足够？

  **仅使用准确率（Accuracy）是不够的。**

  对于智能体而言，智能不仅体现在回答是否正确，还体现在是否能够完成任务、合理规划和适应环境，因此需要从多个维度综合评估。

  常见评价指标包括：

  ### ① 任务完成率（Task Success Rate）

  智能体是否真正完成了用户目标。

  例如：

  - 是否成功规划旅行？
  - 是否成功完成退款？
  - 是否找到正确景点？

  这是评价 Agent 最重要的指标之一。

  ------

  ### ② 准确率（Accuracy）

  回答或决策是否正确。

  例如：

  - 天气是否正确；
  - 推荐景点是否符合条件；
  - 推理结果是否正确。

  这是评价智能体的重要指标，但不能全面反映智能水平。

  ------

  ### ③ 效率（Efficiency）

  智能体完成任务所需的资源，例如：

  - 推理时间；
  - 工具调用次数；
  - Token 消耗；
  - API 调用次数。

  高效的智能体能够用更少资源完成相同任务。

  ------

  ### ④ 推理与规划能力（Reasoning & Planning）

  智能体是否能够：

  - 制定合理计划；
  - 正确选择工具；
  - 在复杂任务中进行多步推理；
  - 根据环境变化动态调整策略。

  这是区别传统 Workflow 和智能 Agent 的重要能力。

  ------

  ### ⑤ 鲁棒性（Robustness）

  面对异常情况时是否能够正常工作，例如：

  - API 调用失败；
  - 用户输入错误；
  - 信息缺失；
  - 工具返回异常。

  优秀的智能体应能够进行容错和恢复，而不是直接失败。

  ------

  ### ⑥ 用户满意度（User Satisfaction）

  最终目标是解决用户问题，因此还应关注：

  - 推荐是否符合用户需求；
  - 回答是否自然、易理解；
  - 用户是否愿意继续使用。

  在实际产品中，用户满意度往往是最重要的评价指标之一。
